# Boost Method

## Purpose

Boost is the redesign layer.

It improves selected screens by combining:

- Audit insight
- Heuristic insight
- the original Albilad DS
- the Commencis Boost component kit
- the taste and direction already established in delivered Boost screens

## Core Rule

Boost is not a blank-page design exercise.

It must behave like a continuation of the delivered Commencis after screens.

## Required Inputs

Before redesigning a screen, review:

1. the current Albilad screen
2. the matching or nearest Boost precedent
3. relevant Audit items for the involved components and foundations
4. relevant Heuristic findings for the screen or similar flows
5. the Boost component kit and guideline pages

## Decision Order

For each meaningful UI region on the screen:

1. Check whether the component or pattern already appears in the Audit.
2. Check whether the screen problem already appears in the Heuristic work.
3. Check whether the same or similar case already appears in existing Boost screens.
4. Check whether the Boost component kit already supports the need.
5. Reuse the existing kit component if possible.
6. If reuse is not enough, extend an existing variant.
7. Only create a new component if variant extension is insufficient.

## Heuristic Fallback Rule

If no explicit heuristic finding exists for the target case:

- infer one using the existing heuristic methodology
- keep it lightweight but rigorous
- use it as redesign guidance

Boost should be able to keep moving even when the heuristic report does not yet cover the exact screen.

## Component Kit Rule

The Boost component kit is the first implementation layer for redesigned screens.

Use it to:

- reuse existing components
- extend variants
- inherit tokens, spacing, typography, and interaction patterns
- maintain consistency with already delivered screens

Do not bypass the kit unless there is a strong structural reason.

## Visual Taste Rule

Taste should come from the delivered Boost work, not from generic inspiration.

The redesign should feel:

- premium but restrained
- clear and trustworthy
- orderly
- modern without trend noise
- consistent with the existing Commencis after screens

## Output Standard

A strong Boost result should:

- solve the real UX problem
- respect DS improvement logic
- preserve accessibility
- scale through components
- look like it belongs to the existing Boost family

## When To Update Other Workstreams

During Boost work:

- add to Audit if a missing DS issue becomes obvious and reusable
- add to Heuristic if a new UX issue is needed to justify the redesign

Boost is allowed to trigger the other workstreams, but should not rewrite them casually.
