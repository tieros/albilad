# Heuristic Method

## Purpose

The Heuristic workstream evaluates existing screens and flows from a UX perspective.

It is product-facing, not library-facing.

## Evaluation Model

Use the same logic already present in the delivered Heuristic Analysis file.

Preserve the existing structure, wording discipline, and severity logic.

When a new screen is reviewed, do not introduce a different heuristic template.

## Baseline Heuristic Set

Use the established ten-heuristic model as the analytical backbone.
If the existing report uses the standard Nielsen framing, preserve that wording and order whenever possible.

Typical coverage includes:

1. Visibility of system status
2. Match between system and the real world
3. User control and freedom
4. Consistency and standards
5. Error prevention
6. Recognition rather than recall
7. Flexibility and efficiency of use
8. Aesthetic and minimalist design
9. Help users recognize, diagnose, and recover from errors
10. Help and documentation

## Deliverable Behavior

Each heuristic review should:

- identify concrete UX issues
- map them to the right heuristic principle
- explain the user-facing impact
- support future redesign work

## Evidence Sources

Review the following before writing:

1. the target Albilad screen or flow
2. existing heuristic findings for related flows
3. similar patterns in the Boost screens
4. the relevant parts of the DS when a usability problem is component-driven

## Continuation Rules

When a new heuristic request arrives:

- check whether similar findings already exist in previous flow reviews
- reuse the same naming and reasoning pattern
- keep the level of detail aligned with the current report
- add new findings only when the issue is materially distinct

## Output Standard

A strong heuristic result should be:

- clear enough to guide redesign
- specific enough to be actionable
- consistent enough to fit the existing report

## When To Escalate Into Audit

If a heuristic issue clearly stems from a design-system problem:

- note the DS dependency
- check whether the issue already exists in Audit
- if not, flag it as a candidate for Audit expansion

Do not force every UX problem into the Audit workstream.
