# Shared Principles

## 1. Continuity Over Novelty

The first job is continuation, not reinvention.

- Continue the Commencis method.
- Continue the Commencis tone.
- Continue the Commencis level of rigor.
- Continue the Commencis visual taste.

## 2. Reuse Before Creation

Always prefer:

1. existing Boost component
2. existing Boost variant
3. extension of an existing component
4. new component creation

New components are the last resort, not the default response.

## 3. Atomic and Scalable Thinking

All UI decisions should support:

- composability
- reuse
- scalability
- accessibility
- maintainability
- token and variant clarity

Avoid creating components that should be variants, wrappers, or compositions.

## 4. Accessibility Is Non-Negotiable

Accessibility must be built into all decisions.

Pay close attention to:

- contrast
- touch targets
- state visibility
- semantic feedback
- readable hierarchy
- error prevention and recovery

## 5. Evidence-Based Recommendations

Recommendations should be grounded in:

1. delivered Commencis work
2. the original Albilad DS
3. the Boost component kit and guidelines
4. strong industry precedents

Use mature systems such as Carbon, IBM, Atlassian, Polaris, Base, Wise, and similar references to support reasoning, not to copy blindly.

## 6. UX Strength Is Required

Heuristic analysis should be strong enough to produce usable redesign guidance even when there is no prior heuristic note for the exact screen.

If a similar finding already exists, reuse its logic.
If not, derive a new finding with the same evaluation discipline.

## 7. Taste Matters

Visual quality is not only about correctness.

Boost outputs must preserve the taste already established in the delivered after screens:

- clear hierarchy
- clean composition
- purposeful spacing
- calm but premium interfaces
- bank-grade clarity
- system-friendly UI

Do not generate generic, trend-chasing, or inconsistent UI.

## 8. Consistency Beats Volume

Do not add findings, heuristics, or components just to make the work feel more complete.

Add only what improves clarity, continuity, or implementation value.

## 9. Separate Method From Execution

- Method files explain what good work looks like.
- Skill files explain how an agent should perform the work.

Do not collapse them into one file.

## 10. Respect The Existing Information Architecture

Audit categories should follow the same logic used in the delivered report and in the original Albilad DS structure whenever applicable.

This is especially important in component domains such as:

- Navigation
- Actions
- Selection Controls
- Forms
- Indicators
- Alerts
- Overlays
- Structure
